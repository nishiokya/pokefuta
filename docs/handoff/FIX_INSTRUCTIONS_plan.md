## P0-1【詳細ページ】「撮ると写真図鑑も埋まる」カードの**進捗数字が欠落**している ★最優先
**現状**: カード内が見出し「🏆 撮ると写真図鑑も埋まる」だけで、その下の進捗ブロックが描画されていない。
**これは案Aの心臓**。数字が無いと「撮る＝何が進むか」が消え、ただのラベルになる。

**あるべき姿**（`reference/screens-AB.jsx` の `AB_Detail` / README §4.1-3）:
見出しの直下に、`--card-2` 地・`radius 12`・`padding 11/13` のブロックを置き、**必ず次の2値を描画**する。

```
[ 北海道 写真図鑑        |   +1     ]
[ 13 → 14 / 50          | シリーズ進捗 ]
```
- 左: `{prefecture} 写真図鑑` ラベ（12px/inkSoft）＋ `current → current+1 / total`。`current+1`(=14) は **terracotta 強調**、矢印と total は inkFaint。数字フォントは Outfit 800/18px。
- 区切り: `width 1 / height 30 / --line` の縦罫。
- 右: `+{seriesProgressDelta}`（green・Outfit 800/18px）＋ ラベル「シリーズ進捗」。
- データ: props `prefectureDex:{current,total}` と `seriesProgressDelta`（README §3.2）。サーバ値が無い場合でも、当該都道府県の写真図鑑進捗から算出して必ず出す。**空表示・非表示は不可。**

## P0-2【マイ旅】写真ゼロのレコードに**投稿依頼CTAが無い** ★最優先
**現状**: 「名古屋市中区・金シャチ横丁…」等、写真の無い訪問レコードが、グレーのアイコン＋希少性タグだけで**ただ並んでいる**。撮影/投稿への導線が一切ない。ここは未投稿300枚超を削る一丁目一番地。
**さらに**: 同一レコードが3回重複表示（要 dedup、P1-2）。

**あるべき姿**: 写真ゼロの各レコード行に、**インライン縮約版 PhotoPrompt** を仕込む（README §4.3「`withHero={false}` の差し込み（縮約版）」）。
- 行の主CTAとして **`📷 写真を追加`**（terracotta、行内に収まるサイズ）を必ず出す。
- 補助に図鑑フック1行：`この旅の写真はまだ0枚 · 撮ると図鑑が埋まる`。
- variant 判定は README §3.1（`photoCount===0` → empty 文言／`>0 && !viewerHasPhoto` → posted 文言）。マイ旅では基本 `empty`。
- 写真ありレコード（豊橋・名古屋の上2件）はそのままでOK。**写真の有無で行の見た目を明確に差を付ける**（写真ゼロ行＝点線サムネ＋追加CTA）。

> 「あとで」を押した場合の挙動も統一: **dismiss（全消し）ではなく collapse（その行だけ畳む）**。`pokefuta.id`／レコード変化で展開状態にリセット（セッション全消えバグの回避。README §5 に準拠）。

## P1-1【マイ旅】達成率（PASSPORT）が主役のまま。**写真図鑑を主役・達成率は小さく**
**現状**: 最上部が「POKEFUTA PASSPORT」＋ 全国80/470・都道府県10/47・ポケモン104/534 の**達成率3指標**が一番目立つ。これは元の戦略メモで「混乱の原因」とされ、達成率は最劣後（Cランク）。今の実装は方針と逆。

**あるべき姿**（`reference/screens-A.jsx` `A_Home` / `screens-PC.jsx` `PC_Home`）:
- **主役を写真図鑑に差し替え**: `PHOTO POKÉDEX / あなたの写真図鑑`。指標は **写真化した訪問 `52/80`** と **未投稿 `28`(terracotta)** を前面に。メーター fill は `linear-gradient(90deg,#e2a015,#bf5640)`。
- 補足コピー「訪れたのに**写真がまだ28枚**。1枚撮るごとに図鑑が埋まります。」
- 「あと2枚で岐阜県をコンプ」カード（trophy リボン＋amberメーター＋写真ゼロ2件の撮るチップ）を写真図鑑の直下に。
- **達成率(全国/都道府県/ポケモン)は撤去はしないが二次情報に降格**：折りたたみ or 小さな副次行に。決して最上部の主役にしない。
- マイ旅の役割は「思い出を見る（写真中心）」。達成率は小さく（戦略メモの `/my-trip` 指針）。

## P1-2【マイ旅】同一訪問レコードの重複表示
**現状**: 「名古屋市中区・金シャチ横丁 宗春ゾーン（東門エリア）2026/5/4」が3件連続。
**対応**: レコード一覧の dedup（同一 pokefuta.id × 同一訪問日でユニーク化）。重複がデータ起因ならクエリ側、表示起因ならレンダリング側を確認。

## P1-3【全体】CTA の重複・冗長
**現状**: マイ旅で `訪問を記録 / スタンプ帳` が画面上部と下部で重複。
**対応**: 一次CTAは1箇所に集約。立ち上げ期の主役は「写真を増やす」なので、固定表示するなら **写真追加系**を優先。
