# ハンドオフ指示書: 写真投稿プロンプト共通コンポーネント（PhotoPrompt）

## 0. これは何か / ゴール
pokefuta.com のログイン後・全画面で、**「写真がまだ無い / 自分が撮っていないポケふた」への写真投稿を促す共通コンポーネント**を実装するための指示書です。

事業フェーズ（SNS×SEO×UGC立ち上げ期）の最重要KPIは **写真投稿率の向上**。未投稿が300枚以上あり、「旅の計画」より「写真を1枚増やす導線」がROI最大、という前提で設計しています。

**採用方針 = 案A（基盤）＋ 案B（フック）の統合：**
- **A＝理屈の土台（ゲーミフィケーション）**: 撮影を「あなたの写真図鑑を埋める」収集体験に再定義。`あなたの図鑑はこの場所 0/1` は写真の有無に関係なく常に成立するので、土台として全状態で効く。
- **B＝感情フック（FOMO/希少性）**: 「一番乗り」。写真の有無で文言を出し分ける（後述の2状態）。

## 1. デザインファイルについて（重要）
このバンドル内の HTML/JSX は **デザインの参照（HTMLで作ったプロトタイプ）** です。**そのまま本番に貼るコードではありません。**
タスクは、これらのデザインを **対象コードベースの既存環境（React/Vue/Next/native 等）と既存のデザインシステム・コンポーネント群を使って作り直す** ことです。まだ環境が無い場合は、プロジェクトに最適なフレームワークを選んで実装してください。

- スタイリングは可能な限り既存のトークン/コンポーネントにマッピングしてください。対応物が無い値だけ、本書のトークン表を使用。
- アイコンは本プロトタイプ独自の簡易ストロークSVG。**既存のアイコンライブラリ**（camera / flag / trophy / star / image / map-pin / route / plus 等）に置き換えてOK。

## 2. 忠実度（Fidelity）
**ハイファイ（hifi）**。色・タイポ・余白・コピーは確定値として扱ってください。レイアウトはピクセル基準で再現し、スタイルは既存ライブラリで合わせ込む方針です。

## 2.5 データモデルの前提（★必読・誤解しやすい）
- **全国マンホール総数 = 470**。コレクションの分母は常に **470**（「80」等の暫定分母を使わない）。
- **訪問 = 撮影 = 1スタンプ**。ユーザーが「訪れた」記録には必ず本人の写真がある。**「訪問したが未撮影」という個人ギャップは存在しない**（`52/80`＝訪問80のうち撮影52、のような表現は誤り）。
  - 個人コレクション数 = 訪問数 = 撮影数 = 例 **80 / 470**（完成率 ≈ 17%）。
  - → **マイ旅（過去の訪問）に「写真を追加」CTAは出さない**。過去の訪問は既に撮影済み。
- **写真投稿KPI（写真0枚 / 一番乗り）は「未訪問マンホール」の話**。誰もまだ訪問=撮影していない場所＝公開写真0枚。だから**写真を増やす導線は「詳細ページ／探す（未訪問マンホール）」に置く**。`PhotoPrompt` の `empty`（写真0枚）はこの未訪問ケース。
- まとめ: *「写真を増やす」＝「新しいマンホールを訪問=撮影する」*。既訪問の再撮影ではない。

## 3. コンポーネント仕様: `<PhotoPrompt>`

### 3.1 状態（variant）の決定ロジック
1ポケふたごとに以下で分岐：

| variant | 条件 | 役割 |
|---|---|---|
| `empty` | `photoCount === 0` | **一番乗り**（写真0枚） |
| `posted` | `photoCount > 0 && !viewerHasPhoto` | **構図を加える**（他者は投稿済み・自分は未投稿） |
| `done` | `viewerHasPhoto === true` | プロンプト非表示（or 控えめな「投稿済み」表示） |

> 重要: `posted` でも **Aの個人図鑑ギャップ（0/1）は残す**。誰かが投稿済みでも「あなたにとっては未記録」で押し続けられる → 全画面で破綻しない。

### 3.2 想定 Props（API例）
```ts
type PhotoPromptProps = {
  variant: 'empty' | 'posted' | 'done';
  pokefuta: {
    id: string;
    title: string;            // 例: "北海道網走のポケふた"
    prefecture: string;       // "北海道"
    area: string;             // "網走"
    photoCount: number;       // みんなの写真枚数
    viewerHasPhoto: boolean;
    rarityTags: string[];     // 例: ["マニューラは全国でここだけ", "網走で唯一のポケふた", "北海道 設置数日本一"]
  };
  prefectureDex: { current: number; total: number };  // 例 {current:13,total:50}（撮影後 current+1 を強調表示）
  seriesProgressDelta?: number;  // 例 +1（シリーズ進捗）
  onPost: () => void;            // 主CTA: 撮影/投稿フローへ
  onRoute?: () => void;          // 経路案内
  onLater?: () => void;          // あとで
};
```

### 3.3 コピー（確定・出し分け表）
| 要素 | `empty`（写真0枚） | `posted`（写真あり・自分未投稿） |
|---|---|---|
| ヒーローbadge（左上） | `🚩 一番乗りチャンス` | `🖼 みんなの写真 {photoCount}枚` |
| ヒーローbadge（右上） | （なし／`未訪問`） | `あなたは未投稿`（terracotta地・白文字） |
| ヒーロー中央 | `0` ＋ `この場所の写真はまだ0枚` ＋ `あなたが最初の記録者に`(terracotta) | 既存写真サムネ（実画像）|
| 見出し直下サブ | （なし） | `あなたの構図で塗り替える`(terracotta, 13px/700) |
| カード上部リボン左 | `🚩 まだ誰も投稿していない` | `🖼 あなたはまだ未記録` |

## 4. 画面ごとのレイアウト仕様（hifi）

設計幅 **390px**（モバイル基準）。上下に共通アプリクロム（ヘッダー / 下タブ）。一部PC考慮（PCでは詳細を2カラム化し、プロンプトカードを右サイドに固定する等で流用可）。

### 4.1 詳細ページ（PhotoPromptの主舞台）
縦並び `flex column / gap 14px`、外周 padding 16px。

1. **ヒーロー**: `height 192px / radius 16px / overflow hidden`
   - `empty`: 点線枠 `2px dashed #cdbf9f`、地は斜めストライプ `repeating-linear-gradient(135deg,#f3ecdc 0 11px,#ece2cd 11px 22px)`。中央に大きな `0`(Outfit 800/40px, #cdbb92)＋テキスト。左上に「一番乗りチャンス」badge。
   - `posted`: 実写真（全面）。左上「みんなの写真 N枚」badge（白90%地）、右上「あなたは未投稿」badge（terracotta地）。
2. **ロケーション＋タイトル**: `📍 北海道 / 網走`(12px/600, ink-faint) → タイトル(M PLUS Rounded 800/21px)。`posted` はサブ「あなたの構図で塗り替える」を追加。
3. **統合カード** `card / radius 18 / border 1.5px #efd9a3 / overflow hidden`:
   - 上部リボン: 地 `linear-gradient(100deg,#fdeae2,#fdf1e6)`、padding 11/14。左にflag(またはimage)アイコン＋短文、右に `0人 › #1`（または `0 /1 図鑑`）。テキストは `white-space:nowrap`。
   - 本体: padding 14。アイコンチップ(28px, 角丸8, 地#fde2c2, trophy/amber-d) ＋ 見出し「撮ると写真図鑑も埋まる」。
   - 進捗ブロック: 地 card-2、radius 12、padding 11/13。左「北海道 写真図鑑 13 → **14** / 50」（14 を terracotta）。区切り線。右「**+1** / シリーズ進捗」（+1 を green）。
4. **補助ピル群**: `flex wrap / gap 6`。ピル = `radius 999 / padding 5×10 / 11.5px / 700`。色種は peach / lavender / mint（下記トークン）。
5. **主CTA**: `width 100% / padding 15 / radius 12`、terracotta地・白文字・`box-shadow 0 2px 0 #a8462f`、M PLUS Rounded 800/16px、アイコン＋ラベル。
6. **副CTA行**: `flex gap 10`。左=経路案内(green地/白)、右=あとで(白地/border line/ink-soft)。各 `padding 12 / 13.5px`。

### 4.2 スタンプ帳ホーム（A基盤の常設プロンプト）
1. **写真図鑑カード**（最上部）: ラベル `PHOTO POKÉDEX`(Outfit/amber-d) → 見出し「あなたの写真図鑑」 → 右に大数字 `52/80`(Outfit 800/26px)。メーター `height 9 / radius 999`、fill = `linear-gradient(90deg,#e2a015,#bf5640)`。補足「訪れたのに**写真がまだ28枚**。1枚撮るごとに図鑑が埋まります。」
2. **「あと2枚でコンプリート」カード**: 上部リボン(地 `linear-gradient(100deg,#fef3d6,#fce8d8)`)に trophy＋「あと2枚で岐阜県をコンプリート」＋右「3/5」。本体に amber メーター(60%)＋写真0枚ポケふた2枚を **空フレーム＋「撮る」ボタン**で横並び。
3. **コレクショングリッド** `grid 4列 / gap 9`: 撮影済=円形プレースホルダ、写真0枚=点線円フレーム＋下に terracotta リボン「📷 あと1枚」。脚注「点線の枠 = 写真がまだ無いポケふた」。

> 他の発火場所（要マッピング）: マイ旅の各記録、投稿完了後のサンクス画面、検索/地図のピン詳細。同じ `<PhotoPrompt>` を文脈に応じ縮約版で差し込む想定。

### `withHero` プロップの使い分け
`<PhotoPrompt>` はヒーローを内包するか切り離すかを `withHero` で切り替える前提（テンプレ参照）。

| 文脈 | 使い方 | 理由 |
|---|---|---|
| **モバイル詳細** | `withHero={true}`（既定） | ヒーロー→本文→カードが1カラムで自然に縦に流れる |
| **PC詳細** | `withHero={false}` ＋ ヒーローは左カラムが描画、カードのみ右 sticky | ヒーローとCTAカードを左右に分離して常時表示するため |
| **ホーム/マイ旅/サンクス等の差し込み（縮約版）** | `withHero={false}`、進捗＋CTAのカードだけを埋め込む | 文脈側に画像があり、プロンプトは行動喚起ブロックのみで足りる |

> 実装メモ: ヒーローの描画責務をどちらが持つかが唯一の分岐。`withHero=false` のときカードは「リボン＋図鑑進捗＋CTA」の自己完結ブロックになり、サイドバー・モーダル・リスト内カードなど任意の器に置ける。variant（empty/posted）判定とコピーは器に関係なく共通。

## 5. インタラクション & 挙動
- 主CTA `onPost`: ネイティブ撮影 or 画像選択 → 投稿フォーム → 投稿完了。完了後、当該ポケふたは `viewerHasPhoto=true` となり variant が `done` に遷移（プロンプトは消える/控えめ化）。
- 進捗数値（13→14, +1, 図鑑52/80）は**投稿成功時に楽観的更新**し、軽いカウントアップ/メーター伸長アニメ（150–250ms / ease-out）。装飾の無限ループは不可。
- 未ログイン時に主CTAを押した場合は**新規登録/ログインへ誘導**（登録増もKPI）。
- `prefers-reduced-motion` 尊重。アニメ無しでも最終状態が見えること。
- ホバー/アクティブ: ボタンは `:active` で 1–2px 沈み（shadow縮small）。

## 6. 状態管理 / データ要件
- ポケふた単位: `photoCount`, `viewerHasPhoto`, `rarityTags`, `prefectureDex{current,total}`, `seriesProgressDelta`。
- ユーザー単位: 写真図鑑（訪問数 / 撮影済み数）、都道府県別の図鑑進捗。
- variant はサーバ値から純粋に導出（3.1）。クライアントで投稿成功イベントを購読し再計算。

## 10. 実装の進め方（推奨）
1. `PhotoPrompt.template.tsx` をコードベースのコンポーネント規約に合わせて移植。
2. トークンを既存デザインシステムへマッピング（無いものだけ §7 を採用）。
3. variant 出し分け（§3.1）をサーバ値から実装 → 詳細ページに配置。
4. 投稿成功イベントで楽観的更新＆ variant 再計算。
5. ホーム/マイ旅/投稿完了など他文脈に縮約版を展開。

優先度: **S=詳細ページの empty/posted ＋ 投稿フロー** → A=ホーム常設プロンプト → B=他文脈展開。案Cはポイント経済の設計が前提のため後段。
